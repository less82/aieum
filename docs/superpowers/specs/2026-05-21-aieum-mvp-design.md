## 1. 제품 정체성

"AI가 찾아주는 완벽한 인연" — 외모·조건이 아닌 **성격/가치관 궁합** 기반으로, 진지한 연애·결혼을 원하는 **2030 세대**용 데이트 매칭 앱.

**4대 차별점**
1. 성향 설문 → **궁합 % + AI 분석 근거** 제시 (스펙이 아닌 성격 매칭)
2. **AI 데이트 코스 A/B안** → 양측이 같은 코스 선택 시 매칭 성공
3. **신뢰·안심**: 휴대폰 본인인증 + QR/GPS 데이트 체크인
4. **관계 빌드업**: 아이스브레이킹, 단계별 정보 공개, 게이미피케이션

---

## 2. 목표와 맥락

- **이번 산출물:** 본 설계 문서 + (후속) 구현 계획 문서. 코드 작성은 다음 단계.
- **MVP 성격:** 투자/데모 프로토타입. 핵심 차별점(성향 매칭·AI 코스·체크인)은 **진짜로 작동**시키고, 비차별 배관(인증·결제·2인 동시접속·QR/GPS 하드웨어)은 **시드/모킹**으로 우회해 일정과 리스크를 최소화한다.
- **선택된 접근법:** "데모 최적화 (B안)" — 차별점은 실제 구현, 주변부는 시드/시뮬레이션.

## 3. 범위 (Scope)

### 3.1 🟢 진짜 구현 (차별점 + 핵심 가치)
- 온보딩 설문(≤10문항, 카드/슬라이더, 프로그레스바) → 응답 저장 — `SCR-ONB-001`
- **성향 점수화 → 궁합 % 계산** (규칙기반 가중 점수, 차원별)
- **AI 분석 근거**(성향차 자연어 설명) + **AI 성향 리포트**(#키워드+요약) — `SCR-MY-001`
- 매칭 화면: 시드 풀에서 **궁합순 추천** + 프로필 카드 + 궁합%·근거 — `SCR-MAT-001/002`
- **AI 데이트 코스 A/B**(LLM 생성 또는 큐레이션 5코스 개인화) → 같은 코스 선택 시 매칭 성공
- 데이트 ING 대시보드: 진행률 게이지 + 단계별 체크인 + 태그형 리뷰 — `SCR-ING-001`
- 마이페이지: 프로필 편집, AI 리포트, 크레딧 잔액

### 3.2 🟡 모킹/시뮬레이션 (데모엔 충분, 비차별)
- 휴대폰 본인인증 → 화면만, 코드입력 없이 통과
- 결제·크레딧 충전 → 결제 UI만, 실제 PG 없이 잔액 증가
- 상대방(2인 동시접속) → 시드 유저가 코스 수락/체크인 **자동 시뮬**
- QR/GPS 체크인 → 버튼 누르면 성공 시뮬(실제 카메라/위치 권한 X)

### 3.3 ⚪ 후순위 (Phase 2 — 데모 제외)
- 실시간 당일 매칭 피드, 실시간 채팅·사진앨범
- 코스 자동생성 고도화(다지역), 매칭 히스토리 시각화
- 게이미피케이션 등급/배지(데모엔 표시만), 보험사 제휴·어뷰징 AI 탐지·응급연락 실연동

### 3.4 MoSCoW 매핑
- **MUST**(전부 데모 포함): 성향 설문, 날짜 선택, 매칭 시작, 매칭자 정보, 궁합률+근거, AI 코스 A/B, 동일코스 매칭성공, 매칭상태 관리, 만남 확인(체크인), AI 분석 리포트.
- **SHOULD**(선별 포함): 코스 리뷰(태그형), 크레딧 시스템(목), 프로필 세부 필터, 신뢰성 인증(모킹). — 당일 즉시 매칭은 Phase 2.
- **COULD**(제외): 설문 게임화 고도화, 실시간 카메오프 추천, 매칭 히스토리, 실시간 채팅, 코스 저장 컬렉션, 등급/배지.

### 3.5 🎬 데모 히어로 동선
가입 → 설문 → 궁합 매칭 추천 → 상대 선택 → 매칭 성공 → 대화창 열기 + AI 코스 A/B 선택 → 데이트 ING 체크인·진행 → 리뷰 → 마이페이지 AI 리포트

## 4. 기술 스택 & 아키텍처

**스택:** Next.js (웹/PWA, 모바일 우선) + FastAPI + Supabase(Postgres + Storage) + OpenAI API

> **데모 구현 노트 (2026-05-21):** 발표용으로는 **백엔드를 보류**하고 **React + Vite 프론트엔드만** 구현했습니다(`frontend/`). 모든 데이터는 클라이언트 시드, AI는 `scripted` 단일 모드(1~2초 "생각하는 척" 로딩)입니다. 궁합 점수만 실제 결정적 계산이고 근거·리포트·코스·후기 요약은 스크립트형입니다. 실동작이 필요하면 `frontend/src/lib/mockApi.ts`를 실제 FastAPI/OpenAI 호출로 교체하면 됩니다.

```
[Next.js PWA]  ──HTTP──>  [FastAPI 백엔드]  ──>  [Supabase Postgres + Storage]
 모바일 우선 UI            도메인 로직 단일 소스        데이터 · 프로필 이미지
 React Query 캐시          (설문채점·매칭·코스·세션)
 목 세션                          │
                                  └──>  [OpenAI API] (근거·리포트·코스, 프롬프트 캐싱)
```

**원칙**
- 프론트는 **FastAPI 한 곳**과만 통신(단일 백엔드 경계). Supabase는 FastAPI가 service-role로 접근 → 데모에서 RLS 복잡도 회피.
- **세션은 목**(휴대폰 인증 모킹) → 데모 유저로 즉시 로그인.
- **데모 무중단 원칙**: 모든 AI 지점에 폴백 → 화면이 절대 빈 채로 멈추지 않음.

## 5. 컴포넌트 (단일 책임 단위)

### 5.1 프론트엔드 (Next.js App Router)
- 라우트: `onboarding/` · `matching/` · `date/` · `profile/`
- UI 컴포넌트: `SurveyCard`, `ProgressBar`, `ProfileCard`, `CompatibilityGraph`, `CourseCompareCard`, `CheckinControl`, `TagReview`, `CreditWallet`, `AIReportCard`, `AIThinkingLoader`(연출 로딩)
- `lib/api`(FastAPI 클라이언트), `lib/session`(목 세션)

### 5.2 백엔드 (FastAPI 모듈)
- `survey`: 질문 스키마 + **점수화 서비스(규칙기반·결정적)**
- `matching`: 시드 풀에서 궁합순 후보 랭킹
- `ai`: 전략 패턴 — `ScriptedAIProvider` / `LiveAIProvider`, `AI_MODE`로 선택 (§7)
- `date`: 세션 상태머신(예정→진행→완료) + 체크인 시뮬
- `review` · `credit`(목) · `profile` · `auth`(목)
- `repository`: Supabase 데이터 접근 계층

## 6. 데이터 모델 (Supabase Postgres 핵심 테이블)

| 테이블 | 핵심 컬럼 |
|---|---|
| `users` | id, nickname, age, region, photo_url, bio, is_seed |
| `survey_responses` | user_id, answers(jsonb), trait_vector(jsonb) |
| `matches` | id, user_a, user_b, course_id, status(pending/success), created_at |
| `date_courses` | id, region, title, steps(jsonb), est_time, est_cost, is_curated |
| `date_sessions` | id, match_id, state, checkins(jsonb), progress |
| `reviews` | session_id, user_id, rating, tags(text[]), ai_summary |
| `credits` | user_id, balance, ledger(jsonb) |
| `ai_scripts` | key(pair/user 식별), kind(reason/report/course), payload(jsonb) — scripted 콘텐츠 & live 폴백 공유 저장소 |

`match_candidates`는 테이블이 아닌 조회 결과(user_id, candidate_id, score, reasons).

## 7. AI 로직

### 7.1 궁합 점수화 (규칙기반·결정적 → 단위테스트 대상)
- 설문 답변 → **차원별 trait_vector**: 외향/내향(E/I), 계획/즉흥(J/P), 정적/액티브, 가치관·연애관 (각 0~1 정규화)
- 궁합% = 차원별 **가중 유사도**. MVP는 **모든 차원을 유사도 기반**(가까울수록 고득점)으로 계산하며, 가중치는 **명시적 상수**로 분리해 튜닝·테스트 용이(예: 가치관·활동성향 高, E/I·J/P 中). 상보성(반대 성향 보상) 매칭은 Phase 2 튜닝 항목으로 보류 — 결정성·단순성 유지.
- LLM 없이도 동작 → 신뢰성·테스트성 확보.

### 7.2 OpenAI 호출 3가지 (모두 폴백 존재)
| 용도 | 입력(구조화) | 모델 | 폴백 |
|---|---|---|---|
| 궁합 근거 설명 | 점수 분해(상위 일치/차이 차원) | gpt-5-nano | 템플릿 문장 |
| AI 성향 리포트 | trait_vector + 답변 | gpt-5-nano | 키워드 규칙 매핑 |
| 데이트 코스 A/B | 지역·활동성향·시간·예산 + 큐레이션 5코스 | gpt-5-mini | 사전생성 A/B |

- 짧은 "구조화 점수 → 자연어 변환"은 nano, 코스 생성만 mini.
- 시스템 프롬프트 + 설문 스키마는 프롬프트 캐싱.

### 7.3 AI 실행 모드 (`AI_MODE` 플래그)
| 모드 | 동작 | 용도 |
|---|---|---|
| `scripted` *(발표 기본값)* | OpenAI 호출 **없음**. `ai_scripts`의 사전 작성 결과를 반환하되 **1~2초 인위적 지연**(`AI_THINK_MS`, 기본 1500) + 프론트 `AIThinkingLoader` "AI가 두 분의 성향을 분석 중…" 연출 | 라이브 시연 — 무위험·결정적·빠름 |
| `live` | 실제 OpenAI 호출(gpt-5-mini/nano), 실패·지연 시 `ai_scripts` 콘텐츠로 자동 폴백 | 실제 동작 증명 |

- `scripted` 모드와 `live` 폴백이 **동일한 `ai_scripts` 저장소를 공유** → 콘텐츠 한 벌만 작성.
- 전환은 환경변수(`AI_MODE`, `AI_THINK_MS`) 한 줄.

## 8. 핵심 플로우

1. **온보딩** (`SCR-ONB-001`): 인증(목) → 설문 N문항 → 채점 → trait_vector 저장 → AI 리포트 비동기 생성.
2. **매칭** (`SCR-MAT-001/002`): 날짜·조건 → 시드풀 궁합순 랭킹 → 후보 카드 → 상대 선택 → 궁합%·근거(AI·캐시) → 코스 A/B → 선택 → (시드 상대 자동 동일선택 시뮬) → **매칭 성공** → date_session 생성.
3. **데이트 ING** (`SCR-ING-001`): 대시보드 → 체크인(QR/GPS 시뮬) → 상태머신(예정→진행→완료) + 진행률 게이지 → 태그형 리뷰 + AI 요약.
4. **마이페이지** (`SCR-MY-001`): 프로필 편집 · AI 리포트 카드 · 크레딧 지갑(목 충전).

---

## 9. 데모 시드 데이터 (발표 동선 고정)

- 데모 본인 계정 1개 + **시드 상대 풀 6~8명** (궁합 다양: 예 92% / 85% / 71% / 58%)
- 1개 지역(예: 성수/홍대) **큐레이션 코스 5개** + 각 페어용 **사전 작성 A/B·근거·리포트**(`ai_scripts`)
- 시드 상대는 코스 선택·체크인을 **자동 시뮬**(매칭 성공·진행 자동 연출)

---

## 10. 에러처리 (데모 무중단 원칙)

- 모든 LLM 지점에 폴백 → 화면이 절대 빈 채로 멈추지 않음.
- 네트워크/백엔드 실패 시 프론트는 마지막 캐시 또는 시드 표시.
- 로딩은 항상 스켈레톤/연출 상태로(끊김 없는 시연).

---

## 11. 테스트 전략

- **궁합 점수화:** 결정적 단위테스트 (TDD 대상 — 가중치·경계값).
- **백엔드:** API 계약 테스트, 상태머신(예정→진행→완료) 전이 테스트.
- **E2E:** 시드 해피패스 1개 (발표 동선 그대로).

---

## 12. 빌드 로드맵 (마일스톤)

1. **M0 기반:** 레포 구조, FastAPI+Supabase 스키마+시드, Next.js 셸·PWA, 목 세션.
2. **M1 온보딩+점수화:** 설문 UI, 채점 서비스(+단위테스트), trait_vector 저장.
3. **M2 매칭+AI 2모드:** 후보 랭킹, 궁합%·근거, 코스 A/B, `scripted`/`live` 모드, 매칭 성공.
4. **M3 데이트 ING:** 대시보드, 체크인 시뮬, 진행률, 태그 리뷰.
5. **M4 마이페이지+마감:** 프로필, AI 리포트, 크레딧 지갑(목), 시드 해피패스 E2E, 발표 리허설.

## 13. 결정된 사항 요약

- LLM = **OpenAI API** (gpt-5-mini: 코스 생성 / gpt-5-nano: 근거·리포트). `BMC.pdf` Cost Structure의 "OpenAI API 비용"과 일치.
- 발표 안정성을 위해 **`scripted` 모드를 기본값**으로, `live`는 실동작 증명용.
- 궁합 점수는 **규칙기반·결정적**, LLM은 "점수를 말로 풀어주는" 보조 역할.
